# スコアアップ施策評価とロードマップ

**対象ノートブック**: `ventilator_pressure_lightgbm.ipynb`
**コンペ**: Google Brain - Ventilator Pressure Prediction
**評価指標**: MAE（Mean Absolute Error）

---

## 現状の把握

| 項目   | 現状                                                                            |
| ---- | ----------------------------------------------------------------------------- |
| 特徴量  | `time_step`, `u_in`, `u_out`, `R`, `C`, `u_in_cumsum`, `u_in_lag(2)` = 7列     |
| モデル  | LGBMRegressor（別カーネルのハイパーパラメータを固定値で流用）                                         |
| 検証方式 | `train_test_split` 80/20（breath をまたいだランダム分割）                                  |
| 主な課題 | ① 特徴量数が少なく時系列情報が不足、② パラメータが現在の特徴量に最適化されていない、③ 同一 breath が学習・検証に混在するデータリークのリスク |

### データの基本構造

このコンペのデータは**肺の RC 回路モデル**をシミュレーションした時系列データ。

```
breath_id | time_step | R  | C  | u_in  | u_out | pressure
    1     |    0.0    | 20 | 10 | 0.083 |   0   |  5.837
    1     |    0.033  | 20 | 10 | 18.38 |   0   |  5.765
    ...
    1     |    2.633  | 20 | 10 | 0.0   |   1   |  0.0
```

- 1 breath = 80 タイムステップ
- `R`（気道抵抗）, `C`（肺コンプライアンス）は breath 内で一定
- `u_in`（吸気バルブ開度）: 連続値、吸気相で変動
- `u_out`（呼気バルブ）: 0=閉（吸気相）/ 1=開（呼気相）
- 吸気相と呼気相で **pressure の挙動が物理的に異なる**

## 施策1: u_out を特徴量から削除

### 評価サマリ

| 項目        | 内容             |
| --------- | -------------- |
| スコアアップ見込み | △ 中（実施方法に強く依存） |
| 難易度       | 低              |
| 優先度       | ★★★            |
|           |                |

### 背景と物理的根拠

`u_out` は呼気バルブの開閉を示すバイナリ変数。

- **`u_out=0`（吸気相）**: 外部からガスが押し込まれ、圧力が上昇。`u_in` の制御に応じて複雑なダイナミクスを示す。**予測困難・学習価値が高い**
- **`u_out=1`（呼気相）**: バルブが開き、圧力は急速に大気圧へ向かって降下。物理的に予測が容易な単調減少パターン

この非対称性のため、呼気相を吸気相と同列に学習させると、**モデルが「圧力が下がる」パターンに引っ張られ**、吸気相の微細な予測精度が下がる。

### 実施方法の比較

| 方法                            | 説明                               | 期待効果    | リスク             |
| ----------------------------- | -------------------------------- | ------- | --------------- |
| A: `u_out` を特徴量から**削除するだけ**   | 列を落として学習継続                       | ほぼなし〜悪化 | 相変移情報が消える       |
| B: `u_out=1` 行を**訓練から除外**して学習 | 吸気相のみで LightGBM を訓練、呼気相は別途ルールで補完 | 中〜大     | 呼気相の予測品質が下がる可能性 |
| C: 吸気・呼気で**モデルを分岐**           | それぞれに専用モデルを学習                    | 大       | 実装コスト高、過適合リスク   |

→ **推奨**: 方法 B。吸気相フィルタリングとして実装し、呼気相は `pressure=0` や前ステップ値で補完する。単に「削除」ではなく「**学習データの質を高める前処理**」として位置づける。

### コードイメージ

```python
# 吸気相のみで学習
train_insp = train[train["u_out"] == 0]
X = train_insp.drop(["id", "breath_id", "pressure"], axis=1)
y = train_insp[TARGET]
```

## 施策2: Optuna によるハイパーパラメータチューニング

### 評価サマリ

| 項目 | 内容 |
|------|------|
| スコアアップ見込み | ○ 小〜中（MAE −0.1〜−0.5 程度） |
| 難易度 | 中 |
| 優先度 | ★★☆ |

### 背景と課題

現在のハイパーパラメータ（例: `learning_rate=0.017`, `min_child_weight=134`）は、**別のカーネルの特徴量セット**に対して最適化された固定値をそのまま流用している。特徴量が変われば最適なパラメータも変わるため、**自分のデータ・特徴量に合わせた再探索が必要**。

### 現状コードの問題点

現在のコメントアウト済み Optuna コードには `subsample/subsample_freq` と `bagging_fraction/bagging_freq` を同時に探索するバグがある（LightGBM は bagging 側を優先し、前者を無視して警告を出す）。本ノートブックでは本番学習側は修正済みだが、Optuna のサーチスペースも整合させる必要がある。

```python
# 修正前（競合パラメータが混在）
"subsample": trial.suggest_float("subsample", 0.4, 1.0),
"subsample_freq": trial.suggest_float("subsample_freq", 0.4, 1.0),
"bagging_fraction": trial.suggest_float("bagging_fraction", 0.4, 1.0),
"bagging_freq": trial.suggest_int("bagging_freq", 1, 7),

# 修正後（bagging のみに統一）
"bagging_fraction": trial.suggest_float("bagging_fraction", 0.4, 1.0),
"bagging_freq": trial.suggest_int("bagging_freq", 1, 7),
```

### 実施上の注意点

1. **特徴量を確定してから実施**: 特徴量が変わると最適パラメータも変わるため、施策1・4 の後に実施する
2. **試行回数**: 最低 50〜100 trials を推奨（10 trials では探索が不十分）
3. **検証方式の改善**: `train_test_split` ではなく `GroupKFold(groups=breath_id)` で評価することで、データリークを防いだ信頼性の高い検証が可能
4. **Early Stopping の活用**: `n_estimators=1000` は固定せず、early stopping で適切な木の本数を自動決定することで過学習を防ぎつつ探索効率が上がる

```python
# GroupKFold を使った Optuna 目的関数の改善例
from sklearn.model_selection import GroupKFold

def objective(trial):
    params = { ... }
    gkf = GroupKFold(n_splits=5)
    scores = []
    for tr_idx, va_idx in gkf.split(X, y, groups=train["breath_id"]):
        X_tr, X_va = X.iloc[tr_idx], X.iloc[va_idx]
        y_tr, y_va = y.iloc[tr_idx], y.iloc[va_idx]
        model = lgb.LGBMRegressor(**params)
        model.fit(X_tr, y_tr,
                  eval_set=[(X_va, y_va)],
                  callbacks=[lgb.early_stopping(50, verbose=False)])
        scores.append(mean_absolute_error(y_va, model.predict(X_va)))
    return np.mean(scores)
```

## 施策3: 特徴量重要度を加味したスコアアップ手法の検討

### 評価サマリ

| 項目        | 内容                    |
| --------- | --------------------- |
| スコアアップ見込み | ◎ 間接的に大（意思決定の土台となる分析） |
| 難易度       | 低                     |
| 優先度       | ★★☆                   |

### 位置づけ

この施策は独立したスコアアップ手段というより、**他の施策の優先順位付けと効果検証に使う分析フレームワーク**として機能する。特徴量重要度の読み方を理解した上で行動に変換することが重要。

### 重要度指標の使い分け

| 指標        | 定義                     | 使用目的                     |
| --------- | ---------------------- | ------------------------ |
| **gain**  | その特徴量が使われた分岐による損失改善の合計 | **予測への実質的貢献度**の評価に使う（推奨） |
| **split** | その特徴量が使われた分岐の回数        | モデルがどの特徴量を頻繁に参照しているかの把握  |

### 分析から得られるアクション

**gain が高い特徴量の場合**
- その特徴量のラグ・リード・ローリング統計など**派生特徴量を優先的に追加**する
- 例: `u_in` の gain が圧倒的に高い → `u_in_lag1`, `u_in_lead1`, `u_in_rolling_mean4` を追加

**gain がほぼ 0 の特徴量の場合**
- **削除候補**として検討（ノイズ特徴量はモデルを不安定にする）
- ただし、gain が低くても交互作用で貢献している場合があるため、削除後の検証 MAE を必ず確認

**split が高く gain が低い特徴量**
- 多用されているが損失改善への貢献が小さい → **過学習の兆候**の可能性あり
- `reg_lambda`（L2 正則化）を強めることで抑制できる場合がある

### 報告資料への活用

施策前後の特徴量重要度を比較することで、「どの特徴量追加が最も効いたか」を視覚的に示せる。**因果関係の説明材料**として研修報告で有効。

## 施策4: 特徴量に lag を追加

### 評価サマリ

| 項目        | 内容                    |
| --------- | --------------------- |
| スコアアップ見込み | ◎ 大（MAE −0.5〜−2.0 程度） |
| 難易度       | 低                     |
| 優先度       | ★★★                   |

### 背景と根拠

LightGBM は決定木ベースのモデルであり、**行内の情報しか見られない**。現在の特徴量では「今この行の `u_in` は何か」しかモデルに伝わらず、「直前の `u_in` からどう変化したか」「数ステップ前の累積吸入量は」といった**時間的文脈が欠落**している。

lag/lead 特徴量を追加することで、LightGBM に**疑似的な記憶**を与えられる。

### 追加候補の詳細

#### u_in 系（最優先）

| 特徴量                  | 計算式                            | 物理的意味                      |
| -------------------- | ------------------------------ | -------------------------- |
| `u_in_lag1`          | `groupby(breath_id).shift(1)`  | 1ステップ前の吸入量                 |
| `u_in_lag3`          | `groupby(breath_id).shift(3)`  | 3ステップ前（初期ベースライン比較用）        |
| `u_in_lag4`          | `groupby(breath_id).shift(4)`  | 4ステップ前                     |
| `u_in_lead1`         | `groupby(breath_id).shift(-1)` | 1ステップ後（LightGBM はリークにならない） |
| `u_in_lead2`         | `groupby(breath_id).shift(-2)` | 2ステップ後                     |
| `u_in_diff1`         | `u_in - u_in_lag1`             | 吸入量の変化率（加速度的情報）            |
| `u_in_rolling_mean4` | `rolling(4).mean()`            | 直近4ステップの平均吸入量              |
| `u_in_rolling_std4`  | `rolling(4).std()`             | 直近4ステップの吸入量のばらつき           |

> **注**: LightGBM は時系列モデルではないため、テストデータにも同じ計算で lead 特徴量を作れる（リーク問題は発生しない）

#### u_out 系

| 特徴量            | 計算式                           | 物理的意味                |
| -------------- | ----------------------------- | -------------------- |
| `u_out_lag1`   | `groupby(breath_id).shift(1)` | 前ステップが呼気相だったか        |
| `u_out_cumsum` | `groupby(breath_id).cumsum()` | 呼気フェーズに入ってからの経過ステップ数 |

#### 位置情報系

| 特徴量               | 計算式                                               | 物理的意味                   |
| ----------------- | ------------------------------------------------- | ----------------------- |
| `step_index`      | `groupby(breath_id).cumcount()`                   | breath 内の絶対ステップ位置（0〜79） |
| `step_norm`       | `step_index / 79`                                 | breath 内の相対位置（0.0〜1.0）  |
| `time_from_start` | `time_step - groupby(breath_id).transform("min")` | breath 開始からの経過時間        |

### 実装上の注意

```python
# breath 境界をまたがないよう groupby 必須
train["u_in_lag1"] = train.groupby("breath_id")["u_in"].shift(1).fillna(0)
train["u_in_lead1"] = train.groupby("breath_id")["u_in"].shift(-1).fillna(0)
train["u_in_diff1"] = train["u_in"] - train["u_in_lag1"]
train["u_in_rolling_mean4"] = (
    train.groupby("breath_id")["u_in"]
    .transform(lambda x: x.rolling(4, min_periods=1).mean())
)
```

## 施策5: LightGBM に適した表形式へデータ整形（ワイドフォーマット）

### 評価サマリ

| 項目 | 内容 |
|------|------|
| スコアアップ見込み | ◎ 大（構造的改善・lag 特徴量の代替・上位） |
| 難易度 | 中 |
| 優先度 | ★★☆ |

### 背景と根拠

#### 現在のロング形式の問題

```
breath_id | step | u_in | pressure  ← モデルは1行ずつしか見られない
    1     |  0   | 0.08 |  5.84
    1     |  1   | 18.3 |  5.77
    ...
    1     | 79   | 0.0  |  0.0
```

→ LightGBM は「ステップ5の情報」を見ながら「ステップ5の pressure」を予測するが、「ステップ0〜4 がどんな値だったか」は lag 特徴量として**手動で与える**しかない。

#### ワイドフォーマットの構造

```
breath_id | R | C | u_in_0 | u_in_1 | ... | u_in_79 | u_out_0 | ... | → target: pressure_0 ... pressure_79
```

- 1行 = 1 breath（全80ステップの情報が1行に集約）
- モデルが **breath 全体のパターンを同時に参照**できる
- ステップ間の交互作用を自然に学習できる
- lag/lead 特徴量が不要になる（全ステップが列として並ぶため）

#### ターゲット変数の扱い

ワイドフォーマットでは `pressure_0〜pressure_79` の 80 列が target になる。

**方法A: マルチターゲット回帰**（`MultiOutputRegressor`）
```python
from sklearn.multioutput import MultiOutputRegressor
model = MultiOutputRegressor(lgb.LGBMRegressor(**params))
model.fit(X_wide, y_wide)  # y_wide: shape (n_breaths, 80)
```

**方法B: ステップごとに 80 個の独立モデル**（計算コスト高）

**方法C: ロング形式に戻してから予測**（構造変換のみの利用）
- ワイドフォーマットで特徴量を作り、ロング形式の予測に使う

### 実装イメージ

```python
# ワイドフォーマットへの変換
def to_wide(df, value_cols, id_col="breath_id"):
    """time_step をインデックスとして pivot"""
    df = df.copy()
    df["step"] = df.groupby(id_col).cumcount()
    wide = df.pivot(index=id_col, columns="step", values=value_cols)
    wide.columns = [f"{col}_step{s}" for col, s in wide.columns]
    # R, C は breath 単位で一定なので別途結合
    rc = df.groupby(id_col)[["R", "C"]].first()
    return wide.join(rc)

X_wide = to_wide(train, value_cols=["u_in", "u_out"])
y_wide = to_wide(train, value_cols=["pressure"])
```

### 施策4（lag 追加）との関係

| 方針 | 説明 |
|------|------|
| ロング形式 + lag 追加（施策4） | 実装が簡単、特徴量を段階的に増やせる |
| ワイドフォーマット（施策5） | 実装コスト中程度、lag 特徴量設計が不要 |

**この2つは排他的なアプローチ**であるため、両方を実験してスコアを比較することを推奨。

## 施策6: モデルを LSTM へ変更

### 評価サマリ

| 項目 | 内容 |
|------|------|
| スコアアップ見込み | ◎◎ 最大（MAE −3〜−8 程度、上位 5% 水準） |
| 難易度 | 高 |
| 優先度 | ★☆☆（Phase 4 で実施） |

### なぜ LSTM が有効か

このコンペのデータは**肺の物理シミュレーション**であり、以下の特性がある。

1. **明確な時間的依存性**: 現在の圧力は直前の吸入量・バルブ状態・前ステップの圧力に依存する（微分方程式）
2. **長期依存性**: breath の初期状態（u_in が高かった / 低かったなど）が後半のステップに影響する
3. **R, C による条件分岐**: 気道抵抗と肺コンプライアンスに応じてダイナミクスが変化する

LSTM はこれらを**アーキテクチャとして本質的に扱える**。LightGBM が特徴量設計で補う部分を、LSTM は自動的に学習する。

### LightGBM vs LSTM の比較

| 観点           | LightGBM            | LSTM / BiLSTM                 |
| ------------ | ------------------- | ----------------------------- |
| 時系列の順序       | 特徴量（lag/lead）で補う    | Hidden state で自然に保持           |
| R, C の条件付き挙動 | 数値特徴量として渡す          | 初期入力として埋め込み可能                 |
| 双方向依存性       | lead 特徴量で擬似的に対応     | BiLSTM で前後方向を同時に学習            |
| 実装コスト        | 低（scikit-learn API） | 高（PyTorch/TensorFlow, GPU 推奨） |
| 学習速度         | 速い                  | 遅い（GPU がないと実用困難）              |
| 解釈性          | 特徴量重要度で可視化可能        | ブラックボックス性が高い                  |
| スコア上限（参考）    | MAE ~0.8〜1.5        | MAE ~0.1〜0.3                  |
|              |                     |                               |

### 推奨アーキテクチャ

上位解法で多用されている **BiLSTM**（双方向 LSTM）構成：

```
入力: [time_step, u_in, u_out] per step + [R, C] as static features
  ↓
R, C → Embedding Layer（または Linear）
  ↓
[Embedding; u_in, u_out, time_step] → BiLSTM(hidden=256, layers=4)
  ↓
出力: pressure per step（回帰ヘッド）
```

- **入力次元**: 1 breath × 80 steps × 特徴量数
- **R, C の扱い**: 連続値のため Embedding ではなく Linear 層で射影するのが自然
- **損失関数**: MAE 直接最適化（`nn.L1Loss()`）または Huber Loss

### 実装の前提条件

- GPU 環境（Kaggle Notebook の GPU を有効化）
- PyTorch または TensorFlow の基礎知識
- バッチ処理のためのデータローダー設計（`breath_id` 単位でバッチを構成）

### 段階的な導入方針

最初から BiLSTM に飛ばず、以下の順で試すことを推奨：

1. **単純 LSTM** → 時系列モデルの動作確認
2. **BiLSTM** → 双方向依存性の改善
3. **BiLSTM + Attention** → 重要なステップへの注目
4. **アンサンブル**（LightGBM + BiLSTM）→ 両者の強みを組み合わせ

## ロードマップ（社内研修向け成果報告）

### Phase 1: 特徴量エンジニアリング（LightGBM 強化）
**目的**: データの時系列性を特徴量として表現し、最も高コスパな改善を行う

- [ ] `u_out=1` 行のフィルタリング実装（施策1）
- [ ] lag-1, lag-3, lag-4, lead-1, lead-2 特徴量の追加（施策4）
- [ ] ローリング平均・標準偏差・diff 特徴量の追加（施策4）
- [ ] `u_out_cumsum`, `step_index`, `step_norm` の追加（施策4）
- [ ] 特徴量重要度で効果を検証（施策3）
- [ ] Kaggle へ提出 → **Phase 0 との MAE 比較**

**成果物**: 改善後のスコア、施策前後の特徴量重要度比較グラフ

### Phase 2: ハイパーパラメータ最適化
