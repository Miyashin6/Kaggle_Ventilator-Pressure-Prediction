## 実施した変更（CHANGELOG #13）

```python
# u_in 差分特徴量
train["u_in_diff1"] = train.groupby("breath_id")["u_in"].diff(1).fillna(0)
train["u_in_diff2"] = train.groupby("breath_id")["u_in"].diff(2).fillna(0)

# R/C 交互作用特徴量
train["R_C_mul"] = train["R"] * train["C"]   # RC積（時定数）
train["R_C_div"] = train["R"] / train["C"]   # R/C比
```

追加後の特徴量数: 7 → 11（`u_in_lag1〜4` を含めると計 13）

---

## 悪化の原因（推定）

---

### 原因1: 差分特徴量が既存の特徴量と線形従属（情報の重複）

`u_in_diff1` は数学的に以下と等しい：

```
u_in_diff1[t] = u_in[t] - u_in[t-1]
              = u_in[t] - u_in_lag1[t]
```

つまり **すでにモデルに与えている `u_in` と `u_in_lag1` から完全に計算できる値**。
LightGBM（決定木）は特徴量間の引き算を直接行えないため、差分を明示的に追加することには意義があるはずだが、
**lag1〜4 が既に揃っている状態**では、モデルはネストした分岐で差分情報を既に近似的に学習できている。

```
既存の木の分岐例:
  if u_in_lag1 < 0.3:
    if u_in > 0.5:   → ここで暗黙的に「差分 > 0.2」を捉えている
```

この状態で `u_in_diff1` を追加すると、**重複した情報が特徴量スペースを占有**し、
本来有用な特徴量への分岐リソースが分散される。

---

### 原因2: R_C_mul / R_C_div がLightGBMに対して冗長

`R` と `C` は既に特徴量に含まれている。LightGBM の決定木は：

```
if R <= 20:
    if C <= 20:   → R*C <= 400 のパターンを木の深さ2でカバーできる
```

のように、**2段階の分岐で積・比を近似できる**。
しかも `R` は 5/20/50 の3値、`C` は 10/20/50 の3値しかなく、組み合わせは **9パターンのみ**。
この9パターンはたった数本の木で完全に分類できるため、`R_C_mul` / `R_C_div` を追加することは
**すでに学習済みの構造を重複して表現する**だけになりやすい。

---

### 原因3: `colsample_bytree` によるサンプリングバランスの崩壊

現在のハイパーパラメータ：

```python
"colsample_bytree": 0.7981   # 列のサンプリング率
```

| 状態 | 特徴量数 | 1木あたりの使用特徴量数 |
|------|---------|----------------------|
| V13（変更前） | 7 | 7 × 0.798 ≈ **5.6 列** |
| V14（変更後） | 11 | 11 × 0.798 ≈ **8.8 列** |

変更前は全13特徴量のうち約5〜6列で木を作ることで**適度な正則化**が効いていた。
変更後は特徴量が増えた分、1木あたりの使用列数が多くなり、**正則化の効果が薄れる**。
Optuna で求めたこの `colsample_bytree` 値は旧特徴量セット（7列）向けに最適化されたものであり、
特徴量を追加した際は**再チューニングが必要**。

---

### 原因4: breath開始行の `fillna(0)` が誤ったシグナルを生成

```python
train["u_in_diff1"] = train.groupby("breath_id")["u_in"].diff(1).fillna(0)
```

各 `breath_id` の先頭行（t=0）は前のステップが存在しないため `diff(1)` が NaN になる。
これを `0` で補完しているが、**実際のu_inはt=0でも正の値（吸気が始まっている）**。

```
breath_id=1:
  t=0.00: u_in=0.08 → diff1 = NaN → 0  ← 実際には「直前比較なし」なのに0として扱われる
  t=0.04: u_in=0.24 → diff1 = +0.16    ← 正しい差分
  t=0.08: u_in=0.51 → diff1 = +0.27    ← 正しい差分
```

全 `breath_id`（約75,000本）の先頭行すべてで `diff1=0, diff2=0` という**人工的なシグナル**が発生する。
モデルはこの「0」を「変化なし」と解釈して学習するが、実際はデータの欠損であるため、
テストデータでは同様のパターンが出ても正しく機能しない（訓練時と意味が異なる）。

---

### 原因5: 施策1（u_out=0フィルタ）との相互作用

施策1では訓練データを `u_out=0`（吸気相）のみに絞っている。
吸気相の最終ステップ（吸気→呼気の切り替え直前）では、`u_in` が急速に減少する。

```
吸気フェーズ末尾:
  t=k-1: u_in=0.40, u_out=0 → diff1 = -0.15 (減少中)
  t=k  : u_in=0.25, u_out=0 → diff1 = -0.15 (減少中) ← 訓練の最終行
  t=k+1: u_in=0.00, u_out=1            ← 訓練から除外
```

訓練データには「吸気フェーズ末尾の急減」パターンが含まれる。
しかしテストデータでは吸気→呼気の境界付近での diff1 の挙動が
訓練時の分布と異なる可能性があり、**train/test の分布ズレ**が生じやすい。

---

## 悪化要因の優先度整理

| 原因 | 影響度 | 備考 |
|------|--------|------|
| ①差分特徴量の情報重複（lag1〜4が既存） | **高** | lag追加後では差分は冗長になりやすい |
| ③colsample_bytreeのミスマッチ | **高** | ハイパラ再チューニングなしに特徴量追加は危険 |
| ④fillna(0)の誤シグナル | **中** | 全breathの先頭行に影響 |
| ②R_C積の冗長性 | **中** | 9パターンしかなく既に学習可能 |
| ⑤施策1との相互作用 | **中** | train/test分布ズレ |

---

## 改善に向けた対策

### 対策A: 特徴量追加後にOptunaで再チューニング

```python
# colsample_bytree を特徴量数に応じて調整
# 特徴量13列の場合、0.4〜0.6程度を探索する
"colsample_bytree": trial.suggest_float("colsample_bytree", 0.3, 0.7)
```

→ **最も効果が高い対策。ハイパラは特徴量セットと一体で最適化する必要がある。**

### 対策B: 差分特徴量の取捨選択（重要度確認）

lag1〜4が既に入っている状態で diff1/diff2 を追加した重要度を確認する。
gain重要度が `u_in_lag1〜4` より著しく低い場合は削除を検討する。

```python
# 重要度の低い特徴量を削除して再実験
features_to_drop = ['u_in_diff1', 'u_in_diff2']  # or ['R_C_mul', 'R_C_div']
X = X.drop(columns=features_to_drop)
```

### 対策C: fillna の方法を変更

breath先頭行の `diff` を `0` ではなく `u_in` 値そのもの（「直前は0だった」という仮定）に変更する。

```python
# breath開始前のu_inを0と仮定した差分
train["u_in_diff1"] = train.groupby("breath_id")["u_in"].diff(1)
train["u_in_diff1"] = train["u_in_diff1"].fillna(train["u_in"])  # 先頭行はu_in自身
```

### 対策D: 追加する特徴量を一つずつ試す（Ablation Study）

```
実験1: diff1のみ追加         → スコア確認
実験2: diff2のみ追加         → スコア確認
実験3: R_C_mulのみ追加       → スコア確認
実験4: R_C_divのみ追加       → スコア確認
実験5: 全部追加              → スコア確認（= V14の現状）
```

一括で追加すると「どれが原因か」が特定できないため、一つずつ検証する。

---

## まとめ

| 観点 | 結論 |
|------|------|
| **差分特徴量** | lag1〜4が既にある状態では差分は冗長。情報の重複が発生しスコアを下げた可能性が高い |
| **RC交互作用** | R・Cが3値×3値=9パターンしかなく、木がすでに近似学習している。追加効果が薄い |
| **ハイパラのミスマッチ** | `colsample_bytree` が旧特徴量数向けに最適化されており、特徴量追加後は再チューニングが必須 |
| **最優先の対策** | Optunaによるハイパーパラメータ再チューニング（施策2）を実施し、新特徴量セットに合わせて最適化する |
