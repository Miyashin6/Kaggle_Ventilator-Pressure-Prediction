# Ventilator Pressure Prediction における LightGBM の選定根拠と実装方針

このコンペで LightGBM を採用した理由と、ノートブックでとっている実装の方針をまとめます。

---

## 1. LightGBM を選んだ根拠

### 1.1 タスクとの相性

| 観点 | 本コンペの内容 | LightGBM が合う理由 |
|------|----------------|----------------------|
| **問題の種類** | 目的変数 `pressure` は**連続値（回帰）** | LightGBM は回帰（`objective="regression"`）を標準でサポートし、MAE を直接最小化できる（`metric="mae"`） |
| **データの形** | 1 行 = 1 タイムステップ。各ステップで `u_in`, `u_out`, `time_step` などが与えられ、そのステップの pressure を予測する | **行単位の表形式データ**として扱えており、勾配ブースティング系（LightGBM）が得意とする「タブラーな入力」と一致する |
| **時系列の扱い** | 本来は 1 breath あたり 80 ステップの時系列 | ラグ・累積・rolling などで**特徴量化して 1 行 1 ベクトル**にしているため、**シーケンスモデルではなく表形式モデル**で解く設計になっている。LightGBM はその「表形式の特徴量」を効率よく扱える |

つまり、「回帰」「表形式の入力」「時系列は特徴量エンジニアリングで吸収」という設計のため、LightGBM が自然な選択肢です。

### 1.2 他のモデルと比べて選んだ理由

| 候補 | 選ばなかった理由（本コンペの文脈） |
|------|-----------------------------------|
| **LSTM / RNN** | シーケンスをそのまま扱えるが、実装・チューニング・学習時間の負荷が大きい。まずは特徴量を作ってタブラーで解く方が、初心者にも扱いやすく再現しやすい。コンペ要件で LightGBM を指定しているため採用。 |
| **線形回帰 / Ridge** | 解釈はしやすいが、`u_in` と pressure の関係は**非線形**になりやすい。木ベースの LightGBM の方が非線形な関係を捉えやすい。 |
| **Random Forest** | タブラーに使えるが、同じ木系なら **LightGBM の方が学習が速く、Kaggle でもスコアが出やすい**ことが多い。 |
| **XGBoost** | 同様に強力だが、**LightGBM の方がデフォルトでも速い**ことが多く、Kaggle の実行時間制限内で試行しやすい。 |

以上から、**回帰・表形式・非線形・速度・コンペ指定**を満たすとして LightGBM を採用しています。

### 1.3 その他の利点

- **学習が速い** … 1 fold あたりの時間が短く、fold 数やイテレーション数を変えて試しやすい。
- **欠損値やカテゴリをそのまま扱える** … 本コンペではほぼ不要だが、他のコンペに流用しやすい。
- **重要度（feature_importance）が出る** … どの特徴量が効いているかの確認に使える。
- **Kaggle でよく使われる** … 情報やノートが多く、再現しやすい。

## 2. 実装の方針

ノートブックでとっている実装の「方針」を、データの扱い・バリデーション・モデル設定・過学習対策・出力の 5 つに分けて整理します。

### 2.1 データの扱い（時系列 → 表形式）

- **breath 単位でソート**  
  `breath_id` と `time_step` でソートし、同一 breath 内で時系列の順序をそろえる。
- **特徴量は breath 内で計算**  
  ラグ・累積・差分・rolling は **breath_id ごとのグループ**で計算し、同じ breath の未来の情報が入らないようにする（リーク防止）。
- **1 行 = 1 ステップ**  
  各ステップを 1 行とし、その行の特徴量からその行の `pressure` を予測する形にする。

これにより、「時系列データ」を「表形式データ」に変換し、LightGBM の入力として渡しています。

### 2.2 バリデーション設計

- **GroupKFold（breath_id でグループ化）**  
  同じ `breath_id` が「訓練」と「検証」に分かれないようにする。
- **理由**  
  1 breath は 80 行で 1 まとまり。通常の KFold だと同一 breath が訓練と検証に割られ、**同じ呼吸の別ステップが検証に漏れる**ため、実質的なリークに近い評価になる。breath 単位で分けることで、**見たことのない breath で検証**し、本番に近い評価ができる。
- **fold 数**  
  実行時間を考慮して 3-fold。精度を優先する場合は 5-fold などに変更可能。

### 2.3 モデル設定（目的・指標・正則化）

- **目的・指標**  
  `objective="regression"`, `metric="mae"` で、コンペの評価指標（MAE）に合わせている。
- **木の複雑さの制限**  
  `num_leaves=20`, `max_depth=6` で木の複雑さを抑え、過学習を抑える。
- **サンプル・特徴量のサブサンプリング**  
  `feature_fraction=0.8`, `bagging_fraction=0.8`, `bagging_freq=5` でバリアンスを下げ、過学習を抑える。
- **early_stopping**  
  検証 MAE が 25 ラウンド改善しなければ打ち切り。無駄なイテレーションを減らし、過学習と実行時間の両方を抑える。
- **n_estimators / learning_rate**  
  実行時間を考慮して `n_estimators=400`, `learning_rate=0.1` に設定。時間に余裕があれば、learning_rate を下げて n_estimators を増やすなどのチューニングが可能。

### 2.4 過学習の判断

- **訓練 MAE と検証 MAE の両方を出力**  
  各 fold で「訓練 MAE」「検証 MAE」「差（検証 − 訓練）」を表示し、平均も表示する。
- **差が大きいほど過学習の可能性**  
  訓練 MAE が検証 MAE よりかなり小さい場合は過学習を疑い、`num_leaves` 削減・`max_depth` 削減・`min_data_in_leaf` 増加・early_stopping の厳格化などを検討する。
- **可視化**  
  各 fold の訓練 MAE と検証 MAE を棒グラフで比較し、fold 間のばらつきや過学習の有無を確認しやすくしている。

### 2.5 テスト予測と提出

- **各 fold のモデルでテストを予測し、平均**  
  `test_pred += model.predict(X_test) / N_FOLDS` で、3 fold の予測の平均を最終提出値にする。これにより、1 つのモデルに依存しすぎず、バリアンスを下げることを狙っている。
- **提出形式**  
  `id` と予測した `pressure` を CSV で保存し、コンペの提出形式に合わせている。

## 3. まとめ

| 項目 | 内容 |
|------|------|
| **選定根拠** | 回帰・表形式・非線形に強く、学習が速く、コンペで LightGBM を採用する要件に合う。時系列は特徴量化してタブラーとして扱う設計のため、シーケンスモデルより LightGBM が適している。 |
| **データ** | breath 単位でソートし、breath 内でラグ・累積・rolling を作成。1 行 1 ステップの表形式で LightGBM に入力。 |
| **バリデーション** | GroupKFold（breath_id）で同一 breath が訓練・検証に分かれないようにし、リークのない評価にする。 |
| **モデル** | regression + MAE、木の複雑さ制限・サブサンプリング・early_stopping で過学習と実行時間を調整。 |
| **過学習** | 訓練 MAE と検証 MAE の差とグラフで過学習を判断し、必要に応じて正則化を強める。 |
| **提出** | 各 fold のテスト予測の平均を提出用の pressure とする。 |

この方針に沿って、ノートブックでは「特徴量作成 → GroupKFold → LightGBM 学習（訓練・検証 MAE の確認）→ テスト予測の平均 → 提出」という流れで実装しています。
